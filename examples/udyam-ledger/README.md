# udyam-ledger — A Shoshin Tutorial

A tiny Go CLI for kirana store owners to log daily sales and print a
bilingual weekly summary. Built end-to-end through the **Shoshin harness**
on **Sarvam 105B** to demonstrate the morning → run → reconvene rhythm.

> **What this is:** a real, working application + an honest transcript of
> what the harness did, what Sarvam produced, and what the human did to
> close the gap. Not a curated demo — a real session captured on
> 2026-04-25.

## What we built

A single-binary CLI:

```
udyam-ledger add-sale <amount-paise> <item>   # Log a sale
udyam-ledger weekly-summary                   # Print this week's summary
udyam-ledger export-csv [out.csv]             # Export to CSV
```

The summary classifies the week as **calm/shaant**, **busy/vyast**, or
**peak/charam** by computing the **digital root** of the total paise and
mapping it to a **three-regime classification** (R1/R2/R3). All math
falls out of an O(1) deterministic rule — no thresholds to tune.

## Setup

```bash
# 1. Initialize Shoshin in the project directory

# 2. Drop in spec.json (or run `shoshin spec` interactively)

```

The spec captures: the project name, one-line goal, primary user (kirana
store owner), target languages (en + hi), Go stack, CLI surface, math
primitives we want, and done invariants.

```
  regime       → internal/math/regime        (+2)
```

Two packages, four files copied. Each is **portable Go code with tests** —
no harness runtime hook. `go test ./internal/math/...` passes immediately.

```go
// In weekly_summary.go:
import digitalroot "udyam-ledger/internal/math/digital_root"

dr := digitalroot.DR(int(totalPaise))      // e.g. DR(27300) = 3
regime := digitalroot.DRToRegime(dr)        // 3 → R3 (peak)
```

This is the `math at the heart` promise: classification is algebraic,
not heuristic. Three sales totaling ₹273.00 → DR = 3 → peak week.
The same code with ₹500 totals → DR = 5 → R2 → busy week. Same algebra,
different output, zero tuning.

or initial tool call before being cut off — they didn't write

### Advance through the state machine

The Feature Done Contract enforces an 8-state lifecycle with evidence
gates per transition:

```
REQUESTED → SCAFFOLDED → MODEL_DONE → VM_DONE → VIEW_DONE → WIRED → VERIFIED → DONE
```

```bash
shoshin features advance add-sale --evidence "Sale type + Add() implemented in internal/add_sale/add_sale.go"
shoshin features advance add-sale --evidence "Subcommand logic + Run() entry point wired in internal/add_sale/add_sale.go"
shoshin features advance add-sale --evidence "Bilingual output formatting in formatConfirmation"
shoshin features advance add-sale --evidence "Wired into main.go switch; binary built and exercised"
shoshin features advance add-sale --evidence "go test ./... — all 5 packages pass: 21 tests..."
```

The VERIFIED gate **rejects** evidence that doesn't mention test/run/verified:

```
✗ VERIFIED requires evidence mentioning a test/run/verification
```

This is the contract doing its job — you can't advance to VERIFIED with
hand-wave evidence.

## Reflection

**What worked well:**
- **PM-driven briefs caught real spec context** the templates would have missed — bilingual, paise (not rupees), CSV headers.
- **Math primitives shipped as portable Go**. Zero runtime dependency on the harness, full test coverage from day one.
- **Feature Done Contract evidence gates fired correctly.** VERIFIED rejected vague evidence and accepted real `go test ./...` output.
- **Stigmergy trail captured honest history** — including where Builders truncated, which is exactly the kind of signal a real reviewer needs.

**What didn't work yet:**
- **Builder `--print` mode is single-shot.** Builders propose or take ONE tool call, then return. They don't write multi-file implementations end-to-end in foundation phase. The honest workflow is: brief → proposal → human finalizes.
- **No automated test runner.** Evidence is human-supplied prose. Future: harness reads `go test` output and synthesizes evidence automatically.
- **State machine prescribes MVVM split** (`MODEL_DONE → VM_DONE → VIEW_DONE`) which is artificial for a CLI app. Stays in the contract for now; the demo just uses the prescribed states for what they map to.

**What's next** (from the Shoshin roadmap):
- **Multi-turn Builders** — drop `--print`, give each Builder a session window to actually finish a feature.
- **Auto-evidence from test output** — replace the `--evidence "..."` flag for the WIRED→VERIFIED transition with `--from-tests`.
- **Williams-bounded parallel dispatch** — currently serial; future runs use `√n × log₂(n)` parallelism.
- **Cross-session MEMORY.md compaction** — Librarian role auto-promotes the daily candidate.
